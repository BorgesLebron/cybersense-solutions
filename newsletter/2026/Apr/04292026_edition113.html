Training Byte

After the Hash: Five Controls That Stop NTLM Theft

The vulnerability:

CVE-2026-32202 doesn't steal your password; it steals your network’s proof of identity. When Windows processes a malicious LNK file, it initiates an automated SMB connection to an attacker-controlled server. This completes an NTLM handshake, handing over your NTLMv2 hash. This hash can be "relayed" in real-time to impersonate you across Active Directory and file shares, or cracked offline to recover your plaintext password. While the April patch closes this specific path, "coerced authentication" is a persistent class of attack that will outlive any single CVE.

Five controls that matter beyond the patch:

  1. Enforce SMB Signing. NTLM relay attacks require unsigned SMB connections. Enforcing SMB signing via GPO on all domain members breaks the relay chain even if a hash is captured.
  2. Block Outbound SMB (Port 445). This vulnerability requires the victim’s machine to reach an external attacker server. Block outbound 445 and 139 at the perimeter firewall to kill the handshake.
  3. Deploy Windows Defender Credential Guard. Use virtualization-based security to isolate NTLM secrets. If the hash is stored in a secure container, LSASS can't be easily scraped or coerced.
  4. Monitor for Anomalous SMB Traffic. Internal machines reaching out to external IPs on port 445 is a high-fidelity indicator of compromise. Alert on these connections in your SIEM immediately.
  5. Restrict NTLM via GPO. Auditing is the first step toward elimination. Use Group Policy to audit NTLM usage, then move to "Restrict NTLM" to prevent its use for remote authentication to untrusted servers.

The hash is the credential. If a hash is coerced, treat it as a full plaintext compromise of that user’s identity.

Living off the XaaS: Nation-States Weaponizing Cloud Infrastructure

The 2026 evolution of tradecraft is "Living off the XaaS" (LotX)—the practice of using trusted cloud platforms like Google Calendar, Azure, and Amazon SES as command-and-control (C2) infrastructure. By hiding encrypted commands in Google Calendar event descriptions or hosting C2 pages on Azure Web Apps, adversaries inherit the "known good" reputation of these providers, rendering IP-reputation filtering and perimeter inspection useless. Final Thought

Yesterday’s KEV entries and this week’s Cloudflare report share a single, uncomfortable truth: the greatest threat isn't sophistication, it’s persistence against known gaps. CVE-2026-32202 succeeded because an initial patch was incomplete; ScreenConnect exploitation persists because remediation was never verified. In an era where 94% of login attempts are bots running at industrial throughput, your defense cannot rely on a "set and forget" patching cycle. The carry-forward for today is simple: Verification is the only true defense. Which of your "patched" vulnerabilities were never actually tested? Which of your remote tools are running versions you haven't confirmed since the last advisory? The attacker's model is throughput—yours must be verification.
